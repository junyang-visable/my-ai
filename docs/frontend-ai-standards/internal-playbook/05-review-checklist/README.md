# 前端 Review 要点（AI 辅助代码）

本目录提供面向 **AI 生成前端代码** 的 Review 清单，用于在合并前统一质量门槛，缓解「同团队内代码质量参差不齐」的问题。

## 为什么需要这份清单

AI 辅助开发会放大个体差异：提示词、上下文、模型版本不同，产出的正确性、类型安全、安全边界可能差异很大。仅靠主观 Review 难以覆盖重复性检查项。本清单把 **合并前必查项**、**建议项** 和 **可选优化** 分层列出，便于评审一致、可追溯。

## 使用方式

### 1. PR Review 时使用

在 Code Review 中打开 `frontend-review-checklist.md`，按章节勾选或对照讨论；🔴 项未通过前不应合并。

### 2. 交给 AI 做合并前自检

将下方指令与 **当前 PR 的 diff / 关键文件** 一并粘贴给 AI，让其按清单逐项核对并标出证据（文件路径、行号或结论）。

**复制以下整段作为提示词：**

```text
请根据以下《前端 Review 清单》对我提供的代码变更做合并前自检。要求：
1. 按清单章节顺序，逐项说明「通过 / 不通过 / 不适用」及简要理由；
2. 对 🔴 项必须给出明确结论；不通过须指出具体位置与修复建议；
3. 对 🟡、🔵 项可汇总建议，标注是否建议在本次 PR 处理；
4. 若信息不足（例如看不到 API 文档），请列出需要我补充的材料。

--- 清单正文开始 ---
（在此粘贴 frontend-review-checklist.md 的全文）
--- 清单正文结束 ---
```

## 严重级别说明

| 级别 | 含义 | 合并策略 |
|------|------|----------|
| 🔴 **阻塞** | 必须在合并前修复或给出经团队认可的例外说明 | 默认 **不允许合并** |
| 🟡 **建议** | 影响可维护性、一致性或体验，可登记技术债后续处理 | **可合并**，建议在 issue / 后续 PR 跟进 |
| 🔵 **优化** | 锦上添花，无阻塞风险 | **可选**，按排期处理 |
