# 页面 / 组件重构 Prompt 模板

## 使用场景

对**已有**页面或大型组件做结构性整理：拆分文件、抽取 composable、消除重复请求逻辑等，且**不改变对外行为**。适用于单文件过长（如 >300 行）、职责混杂、重复逻辑多等维护问题。

---

## Prompt 模板

```markdown
## 上下文

- 请遵循团队 Repowiki 中的开发与代码规范（参考：[替换：Repowiki「技术规范」或 dev standards 路径/链接]）。
- 目标文件：`[替换：待重构文件路径，如 pages/foo/bar.vue 或 components/baz/LegacyPanel.vue]`
- 当前技术栈：Vue 3、TypeScript、Nuxt 3、Pinia（若相关）。

## 任务

对上述文件进行重构，需解决的具体问题（可多选并补充细节）：

- `[替换：例如：单文件 >300 行，需按「展示 / 数据 / 表单」拆分子组件]`
- `[替换：例如：重复的列表请求与分页逻辑，抽取为 composable]`
- `[替换：例如：魔法数字与内联业务规则，集中到常量或小型 util]`

若你判断需要**拆分为多个新文件**，请先在回复**前半部分**给出**拆分计划**（文件列表、每个文件职责、对外仍由谁导出），**待我确认后再输出完整代码**（若我已在消息中写明「已确认拆分方案，可直接输出代码」，则跳过确认步骤直接交付）。

## 约束

1. **不得改变现有 Props / Emits 的对外接口**（名称、类型、语义、默认值）；若原组件无 Props 则保持无 Props。内部实现可调整。
2. **不得改变视觉输出**：布局、间距、字号、颜色、响应式断点表现应与重构前一致（允许 class 名或 DOM 层级在不影响渲染结果的前提下调整，但默认应最小化 DOM 变更）。
3. **不得改变业务逻辑**：条件分支、接口调用时机、校验规则、埋点参数等与产品行为相关的逻辑须与重构前等价；仅允许「搬家」与「抽取」，不允许「顺手优化」导致行为变化。
4. 若必须做极小行为修正才能编译通过，请在输出中**单独高亮说明**并给出理由；否则禁止夹带无关修改。
5. 遵守项目目录约定：composables、components、utils 放置路径与现有代码一致。

## 输出要求

1. **重构后的完整文件内容**（所有被修改或新建的文件，每个文件用路径标题区分）。
2. 若拆分了文件：**新建 / 移动文件清单**（路径列表即可）。
3. **改进说明**：用条目列出「做了哪些结构化改进」（如：哪段逻辑移入哪个 composable），便于 Reviewer 对照 diff 验收。
4. **自检**：用一两句话说明如何验证「界面与行为未变」（例如：对比关键交互路径、原有 E2E / 手工步骤）。
```

---

## 使用说明

- **先列问题再贴代码**：在「任务」里把痛点写具体（行数、重复片段行号、截图或录屏链接可选），模型拆分更准确。
- **强制「行为不变」时**：可在对话中附加「验收：与 main 分支该文件行为一致」或贴上关键用例描述。
- **大 diff 分步提交**：若输出过大，可要求模型按「先抽 composable → 再拆子组件」分两轮生成，降低单次错误率。
- **与产品确认边界**：若重构涉及「疑似死代码」删除，务必在真人确认后再写入 Prompt，避免误删仍在使用的分支。
