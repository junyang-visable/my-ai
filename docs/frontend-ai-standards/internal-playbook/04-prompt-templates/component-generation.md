# 组件生成 Prompt 模板

## 使用场景

需要从零编写新的 **Vue 3 单文件组件（SFC）**，技术栈为 **Vue 3 + TypeScript + Nuxt 3**，使用 `<script setup>`。适用于列表项、表单控件、展示卡片等可复用 UI；生成结果需可直接放入仓库并符合分层与 i18n 约定。

---

## Prompt 模板

```markdown
## 上下文

- 技术栈：Vue 3、TypeScript、Nuxt 3，组件使用 `<script setup lang="ts">`。
- UI 与基础库：[替换：例如 Element Plus / Naive UI / 项目自研组件库名称]；图标与布局遵循现有页面惯例。
- 请参考团队 Repowiki 中的组件文档（路径或章节：[替换：Repowiki 组件文档链接或相对路径说明]），风格与命名与现有代码一致。
- 参考实现（可选但推荐）：[替换：类似组件的文件路径，如 `components/foo/BarCard.vue`]。

## 任务

在路径 `[替换：目标路径，如 components/business/UserProfileCard.vue]` 新建组件 `[替换：组件名，PascalCase]`。

功能与接口设计：

- Props：`[替换：列出 Prop 名、类型、是否必填、默认值与说明]`
- Emits：`[替换：事件名与 payload 类型]`
- 行为 / 交互：`[替换：简述展示逻辑、用户操作、与父组件的数据流]`

若需插槽，请说明：`[替换：具名插槽 / 默认插槽及用途]`

## 约束

1. **禁止在组件内直接发起 HTTP 请求**（不使用 `fetch`、`$fetch`、`axios` 等）；数据一律通过 Props 或 Pinia（若项目约定允许）由上层注入；本模板默认仅 Props + emit。
2. **禁止硬编码面向用户的文案**：使用 `vue-i18n`（`useI18n`）或继续通过 Props 传入已翻译文案；若新增文案，在输出中列出建议的 i18n key。
3. 样式使用 **`<style scoped>`**（或项目约定的 CSS Modules）；**不要引入仓库中不存在的 npm 包或内部路径**。
4. TypeScript：为 Props / Emits 提供明确类型；避免 `any`；导出类型若对父组件有用可 `export type`。
5. 可访问性与语义：在合理范围内使用语义化标签与 `aria-*`（若设计稿无特殊要求则保持简洁）。

## 输出要求

1. **完整的 `.vue` 单文件组件源码**（template / script / style 齐全）。
2. **父组件中的使用示例**（展示 Props、emit 监听、必要时 `v-model` 或插槽）。
3. 若使用了 i18n：**列出新增或复用的 key 清单**（含命名空间与中文/英文占位文案建议）。
4. 简要说明 **与参考组件的差异**（若有），便于 Code Review。
```

---

## 效果优化建议

- **复杂组件先定接口**：Props / Emits / 插槽较多时，可先单独开一轮对话只输出 TypeScript 接口与简要设计说明，确认后再生成完整 SFC，减少大面积返工。
- **务必提供对标组件**：贴一个结构相近的现有组件路径或片段，模型更容易对齐 import 顺序、组合式函数用法与样式习惯。
- **路径与命名一次性写清**：Nuxt 自动导入规则下，组件目录层级影响引用名；在「任务」中写明完整相对路径可避免生成到错误层级。
- **UI 库版本敏感**：若项目锁定 major 版本，在「上下文」中写一句版本号或文档链接，降低 API 用法幻觉风险。
