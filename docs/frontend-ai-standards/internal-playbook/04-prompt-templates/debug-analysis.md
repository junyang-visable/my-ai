# 缺陷分析与修复 Prompt 模板

## 使用场景

线上或本地出现的 **Bug 排查与修复**：根据现象、复现步骤与报错信息，定位根因并给出最小修改；适用于逻辑错误、异步竞态、类型疏忽、环境差异等问题。

---

## Prompt 模板

````markdown
## 上下文

- 相关源码（请粘贴完整函数 / 组件或明确文件路径并粘贴关键片段）：

```vue
[替换：粘贴相关 .vue / .ts 代码]
```

- 运行环境：`[替换：浏览器与版本 / Node 版本；dev 或 production]`  
- 近期变更（可选）：`[替换：相关 PR、commit 或猜测的回归点]`

## 问题

请基于以下信息分析并修复。

**现象：**  
`[替换：用户看到的问题描述]`

**预期行为：**  
`[替换：正确情况下应该怎样]`

**复现步骤：**  
1. `[替换：步骤一]`  
2. `[替换：步骤二]`  
3. `[替换：…]`

**报错信息（如有）：**

```
[替换：完整堆栈或控制台错误文本]
```

## 约束

1. **只修复上述描述的问题**；禁止顺带重构、格式化无关代码或「优化」未提及模块。
2. 若修改多个文件，请在输出中 **按文件逐一解释** 每处变更的目的（每文件 1-3 句）。
3. 若存在多种修复思路，请 **列出备选方案（至少两种若适用）**，并明确 **推荐方案及理由**（可维护性、风险、与现有模式一致性）。
4. 不引入新依赖；不放宽类型仅为消除报错 unless 能证明原类型错误且修复与问题相关。
5. 若信息不足，请 **列出需要补充的最少信息**（如某接口返回值），而不是猜测业务。

## 输出要求

1. **根因说明**：用 1-3 句话概括技术原因（避免只复述现象）。
2. **修复内容**：以 **diff 格式** 展示所有代码变更（或按文件分块的清晰 before/after）。
3. **验证步骤**：给出 Reviewer 或本人可执行的 **确认步骤**（与「复现步骤」对应，说明预期已恢复正常）。
4. （可选）**回归风险提示**：一句话说明是否建议补测试或监控。
````

---

## 效果优化建议

- **堆栈要全**：含文件名与行号时，模型定位更快；可隐去敏感 token。
- **最小复现仓库或片段**：若 Bug 依赖特定数据状态，贴 JSON 或 seed 步骤。
- **先分析后改**：可要求模型「第一轮只输出根因与方案对比，确认后再给 diff」，适合高风险路径。
- **异步与缓存**：涉及 `watch`、`onMounted`、SWR 时，在「现象」中写清「第几次进入页面」「是否快速切换 tab」等时序信息。
