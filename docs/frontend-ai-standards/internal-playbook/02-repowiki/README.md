# RepoWiki（前端版）

RepoWiki 是面向前端仓库的**结构化项目知识库**：用固定章节与表格把「项目是什么、怎么写、路由与组件、状态、业务、怎么测」写清楚，让 AI 与新人都能**直接检索、对照执行**，而不是在长文中猜测意图。

## 三条写作原则

| 原则 | 含义 | 实践要点 |
|------|------|----------|
| **结构化 > 叙述** | 优先表格、列表、代码块，少大段散文 | 同一类信息用同一表格列；可扫描、可 diff |
| **约束 > 描述** | 写清「必须 / 禁止 / 默认」，而不只是「我们通常」 | 用可执行的规则（路径、命名、环境、审批） |
| **背景 > 定义** | 说明**为什么**这样定，减少 AI 误推 | 链接业务、历史坑、性能或合规原因 |

## 目录结构

```text
docs/frontend-ai-standards/internal-playbook/02-repowiki/
├── README.md                          # 本说明：原则、树、用法、验收
└── templates/
    ├── INDEX.template.md              # 知识索引与 AI 阅读顺序
    ├── 01-项目概述.template.md
    ├── 02-技术规范.template.md
    ├── 03-路由与页面结构.template.md
    ├── 04-组件库文档.template.md
    ├── 05-状态管理.template.md
    ├── 06-业务知识.template.md
    └── 07-测试用例.template.md
```

实际落地到业务仓库时，将填好的 Wiki 放在项目根目录的 `repowiki/` 下，并保持与 `templates/` 中章节一一对应，便于同步更新。

## 使用步骤（5 步）

1. **复制模板**：将 `templates/` 下各 `.template.md` 复制到目标仓库根目录的 `repowiki/` 下，去掉 `.template` 后缀（或按团队规范命名）。
2. **填写占位符**：将所有 `[替换]`、`[TODO: …]` 替换为真实信息；无此项则删除行或写「不适用」。
3. **维护 INDEX**：在 `INDEX` 中更新链接与摘要，保证与 01–07 文档标题一致。
4. **评审与冻结**：由前端负责人（或架构代表）做一次完整性检查，确认约束与示例与代码一致。
5. **随代码迭代**：路由、Store、组件 API、业务术语变更时，同步改 Wiki；PR 可把「是否更新 RepoWiki」纳入自检。

## 验收标准

- **核心仓库 `main` 分支必须有一份完整 RepoWiki**：至少包含与模板对应的 7 类文档（或合并为等价章节，但信息不缺失），且 `INDEX` 可导航到全部章节。
- 项目概述、技术规范、路由、组件、状态、业务、测试中的**约束与示例**与当前代码行为一致，无大量过期 `[替换]`。
