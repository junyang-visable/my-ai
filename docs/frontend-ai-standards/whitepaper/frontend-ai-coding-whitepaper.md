# 前端 AI 辅助开发工程实践

> Visable AI 技术白皮书 — 前端篇

## 引言：从「使用 AI」到「工程化 AI」

工程团队已普遍把 AI 用于日常编码；一旦进入真实交付链路，失败方式却高度相似：输出「看起来对」却偏离约束、吃不透业务语境、在边界与安全细节上留洞。将症结简单归为「模型不够强」，往往掩盖真正的短板——缺少把 AI 当作**受约束的生产要素**来运营的方法。模型会持续升级，而**约束工程**（流程、规范、门禁）与**上下文工程**（可检索、可验证的项目知识）才是稳定性的底盘。工程化 AI 追求的不是单次惊艳的生成，而是可复现、可审计、可协作的人机节奏。下文从人机边界、上下文沉淀与流程约束三方面，整理我们在前端侧的实践经验，供同行交流与复盘。

## 一、核心理念：人机协作的边界

### 1.1 AI 的能力边界

在界面与工程实现上，AI 尤其胜任**输入输出边界清晰**的任务：按接口契约补全调用与状态、在既定风格下调样式与布局，以及样板代码、测试骨架、文档结构等重复劳动——当一件事能写成「可查的规范 + 可对齐的示例 + 明确的完成定义」时，初稿产出往往很快。反之，复杂业务取舍、跨版本历史决策、身份与权限等敏感路径，仍须由工程师担任定义者与守门人；实现路径可以多给模型试探，**谁为最终行为背书**必须没有歧义。把边界写进共识，不是为了限制创新，而是把创造力放在真正需要判断力的环节。

### 1.2 代码所有权

我们坚持一条原则：**凡进入主分支的代码，无论是否经 AI 辅助编写，其质量与后果一律由提交者承担。** Review、测试与发布责任不因「机器写过」而减轻。AI 是加速器，不是责任转嫁的借口；架构一致、可读性、安全与可维护性仍由工程师兜底。这反过来约束用法：提示里写清边界、合并前按标准审视产出、发现系统性偏差时改规范与知识库，而不是堆个案补丁。缺陷与事故追溯时，讨论焦点应是流程与上下文是否充分，而非把问题推给工具。权责清楚了，才谈得上稳定的「人机协作纪律」，而不是游离于流程之外的「个人与模型的默契」。

## 二、上下文工程：让 AI 懂你的项目

### 2.1 问题根因

通用大模型并不携带你方代码库的私有记忆；会话之间也不会自动延续隐性上下文。若仅依赖临时粘贴的几段代码与口头描述，模型只能做「合理猜测」，而猜测在工程上等同于风险。更棘手的是，口头上下文会随人员流动与迭代而漂移，未落盘的决策最终只在聊天记录里「曾经正确过」。上下文工程的核心，是把**项目特有的决策、约束与结构**转化为机器可读、人类可维护的文本资产，使每一次协作都站在同一份事实来源之上。它不是把文档写得漂亮，而是把文档写得**可被检索、可被引用、可被校验**——让 AI 与人对齐同一套**仓库里的约定与事实**。

### 2.2 项目知识库（RepoWiki）

我们采用仓库内结构化知识库（RepoWiki）承载前端上下文：文档按类型拆分，避免长篇叙事淹没关键约束；写法遵循「AI 可读」原则——多用标题层级、列表与**可浏览的目录结构**呈现约束，减少隐喻与口号式描述。典型落盘可参照下例（项目根目录 `repowiki/`）：

```text
repowiki/
├── INDEX.md                     # RepoWiki 索引与 AI 阅读顺序
├── 01-项目概述.md               # 模块边界、技术选型摘要、与上下游关系
├── 02-技术规范/               # 栈版本、命名与目录约定、性能基线
│   ├── 技术栈.md
│   ├── 开发规范.md
│   └── 性能基线.md
├── 03-路由与页面结构.md         # 信息架构、权限与守卫、关键导航路径
├── 04-组件库文档.md             # Props、事件、插槽等结构化说明，便于对照生成
├── 05-状态管理.md               # Store 划分、数据流与副作用边界
├── 06-API文档/                  # 请求封装、鉴权、模块列表与错误码；生成接口调用前必查
│   └── 接口约定.md              # 可并列 Swagger / Apifox 导出或 OpenAPI JSON
├── 07-业务知识/                 # 核心流程与领域概念，解释「为何如此设计」
│   ├── 领域概念.md
│   └── 核心流程.md
└── 08-测试用例/                 # 关键场景、期望行为与回归关注点
    └── 测试用例.md
```

RepoWiki 并非一次性产物——它需要与代码同源演进。**每次功能迭代合并后，新路由、新 Store、新接口约定、新业务规则必须同步回写到 RepoWiki**，否则下一次 AI 辅助仍将在过时的上下文中运作。当知识库与代码保持同步时，AI 辅助修改才具备「就地取证」的基础，而不是在真空中编造。

### 2.3 AI 配置规范（.cursorrules）

在编辑器侧，我们通过项目级规则文件（如 `.cursorrules`）把**风格、禁区与协作习惯**固化为默认上下文。其典型覆盖范围包括：语言与框架偏好、组件与样式约定、测试与类型检查的期望、对安全与隐私的硬性约束（例如敏感信息不得写入提示或日志）、以及希望 AI 默认遵守的交互边界（何时应拆分任务、何时应请求更多上下文）。规则应与知识库相互引用：前者定义行为边界，后者提供领域事实，二者缺一都会出现「守规矩但不懂业务」或「懂业务但越界」的失衡。规则文件的价值在于减少「每次对话重复声明」的摩擦，并把团队共识前置到生成之前。它是上下文工程的延伸：不仅告诉模型「项目里有什么」，也明确「在这里应该怎样工作」。

## 三、约束工程：四支柱协同框架

将 AI 有序嵌入前端交付链路，需要四个支柱协同驱动，缺任何一个都会导致流程走样：

| 支柱             | 角色       | 一句话说明                                     |
| ---------------- | ---------- | ---------------------------------------------- |
| **OpenSpec**     | 定方向     | 实现前产出结构化 Spec，Spec 评审通过后才动代码 |
| **Superpower**   | 带节奏     | 接管实施全程：探方案 → 写计划 → 执行 → 验收    |
| **RepoWiki**     | 提供知识库 | AI 在三个关键节点读取项目知识，避免臆测        |
| **.cursorrules** | 纪律约束   | 命名、禁用 API、流程纪律在每次对话中自动生效   |

## 3.1 Spec-Driven Development（SDD）与 OpenSpec

若缺少 upfront 的规格说明，AI 编码容易陷入「看似完成、实则偏离」的陷阱：界面像样，状态与异常路径却未对齐需求。Spec-Driven Development 强调在动手写实现之前，先把**可讨论、可确认的规格**落到文档中，再驱动编码与验收。我们采用 OpenSpec 作为规格载体：在功能开始前产出包含功能范围、数据模型、交互逻辑与技术约束的结构化文档，经人工评审确认后，才将控制权交给实施环节。OpenSpec 提供跑道，跑道清晰，生成才可控。

标准节奏如下：需求到达 → 读 RepoWiki 核实现有约束 → 产出 Spec → 人工评审确认 → 进入实施阶段。**Spec 确认前不动代码**是 SDD 的核心纪律。

## 3.2 Superpower：AI 驱动的实施节奏

OpenSpec 确认后，实施全程由 Superpower 框架接管，确保每一步都有结构：

- **brainstorming**（方案不明时）：先读 RepoWiki 了解现有模块，提 2–3 种方案，产出设计文档
- **writing-plans**：读 RepoWiki 技术规范与路由，生成含文件清单的分步计划
- **subagent-driven-development / executing-plans**：按计划逐步执行，`.cursorrules` 全程约束
- **verification-before-completion → finishing-a-development-branch**：验收通过后提 PR

每个阶段的 RepoWiki 读取并非形式——它保证 AI 在制订计划前、执行前都站在当前项目的真实约束上，而不是只靠当次对话里零星记忆或自行猜测。

## 3.3 Definition of Done

**代码合并不等于完整交付**。一次完整改动的闭环定义是：代码合并 **+** Specs 同步更新 **+** RepoWiki 回写新路由、新 Store、新接口约定与新业务规则。这一定义需写入团队 DoD（Definition of Done），否则上下文资产会随迭代加速腐烂，下一轮 AI 辅助的质量基线将随之下滑。

## 3.4 代码 Review 标准

AI 产出在 Review 维度上有其典型风险：幻觉式 API 或依赖、类型宽松导致的隐性契约漂移、快乐路径覆盖而忽略边界、以及安全与权限上的「看起来对」。我们采用**分级检视**：阻断项聚焦在正确性、安全与合规——必须与需求与规格一致，敏感路径与数据流需可证伪；建议项聚焦在可读性、一致性与长期维护成本——命名、抽象粒度、重复模式等可在合入后持续打磨。分级的好处是既不因完美主义阻塞交付，也不让高风险问题借「AI 写的」蒙混过关。Review 标准与 SDD、知识库联动：规格与文档越是可靠，Review 越能从「猜意图」转为「对照验收」。

## 四、落地路径与角色分工

## 4.1 落地优先级

规范不宜一次性全量铺开，我们建议按以下顺序推进：

1. **先夯实配置与知识库**（`.cursorrules` + RepoWiki）——工具与知识就绪是一切的前提
2. **推广 SDD 节奏**（OpenSpec + Superpower）——节奏稳定后质量可预期
3. **收紧 Review 门禁**（Review Checklist）——流程成熟后用门禁保底
4. **扩充 Prompt 模板库**——随能力建设并行推广

## 4.2 项目 Owner 职责

- 使用 `init-ai-standards` skill 一键初始化 `.cursorrules`、`repowiki/` 与 `specs/`
- 评审并完善 RepoWiki 中的业务知识与路由章节（需人工填写的领域知识）
- 将「合并后同步 Specs + RepoWiki」写入团队 Definition of Done

## 4.3 开发人员工作流

1. 接到新需求 → 先确认 Spec 是否存在（`specs/` 或 `openspec/changes/`）
2. **有 Spec** → Superpower `writing-plans` 写计划 → 执行 → 验收
3. **无 Spec** → 先走 OpenSpec 生成 Spec，评审通过后再开始
4. 合并后 → 同步 `repowiki/` 与 `specs/`

## 五、验收标准

将上述实践视为「可用」的最低限，建议在合入与版本发布前逐项核对（可按团队裁剪，但不宜删除与归因、可追溯相关的条目）：

- **规格与代码一致**：功能改动所依据的 Spec（`specs/` 或 `openspec/changes/`）已评审确认，实现与 Spec 所述范围、数据与交互一致；合并后 Spec 已同步更新，无「代码已改、规格仍旧」的漂移。
- **RepoWiki 可取证**：`repowiki/` 中 `INDEX.md` 可导航至全部约定章节；本次改动涉及的路由、Store、接口约定、业务规则等已回写对应文档，**AI 与人工均可据文档复现调用与行为**，而非依赖口头或聊天记录。
- **纪律与门禁生效**：仓库根目录 `.cursorrules` 就绪且含 SDD / 协作相关条款；PR 已对照团队 Review Checklist 自检，**阻断项**（正确性、安全、合规、与 Spec 对齐）均已闭环或显式豁免并有记录。
- **完成的定义（DoD）闭环**：除代码合并外，**Specs + RepoWiki** 的同步已作为本次改动的 DoD 一部分完成；若项目约定自动化测试或手工验收清单，其结果可追溯（报告、截图或 CI Job）。

满足以上条目，意味着人机协作产物具备**可验收、可事后追溯、可交给下一位开发者或下一轮辅助工具继续使用**的最低可信度；未满足则宜在合并说明中标注缺口与补救计划，避免把技术债伪装成「AI 已交付」。

## 六、持续演进

模型与编辑器能力仍在更替，**没有写一次就永远适用的规范**；长期有用的是**能随项目修订的流程与条文**，而不是某一版界面截图或零散 Prompt。

我们建议把握三条节奏：

1. **先更新文档与规格，再扩大改动范围**：业务或架构一变，应先改 `repowiki/` 与 Spec，再交给 AI 做大范围编辑；若文档与规格滞后，改动面越大，与真实约定偏离越快。
2. **把重复出现的问题写回制度**：每类典型差错（错接口、鉴权遗漏、合入后文档未更新等）应对应补齐**规格模板、`.cursorrules` 条款或短案例**，避免后来人只靠翻聊天记录还原原因。
3. **案例以小为贵**：写清「如何复现、怎样修好」的短文，往往比长篇复盘更易照做；对外交流也侧重做法与教训，少罗列工具名。
