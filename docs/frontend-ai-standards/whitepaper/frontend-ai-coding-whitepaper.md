# 前端 AI 辅助开发工程实践

> Visable AI 技术白皮书 — 前端篇

## 引言：从"使用 AI"到"工程化 AI"

过去一两年，工程团队普遍经历了从好奇试用到日常依赖的转变：智能补全、对话式改码、批量重构似乎触手可及。然而，一旦将 AI 嵌入真实交付链路，失败模式会高度一致地浮现——输出看似合理却偏离约束、对业务语境一知半解、在边界条件与安全细节上留下隐患。将问题简单归咎于「模型不够强」往往掩盖了更本质的短板：我们缺少把 AI 当作**受约束的生产要素**来运营的方法。模型能力会迭代，但**约束工程**（流程、规范、门禁）与**上下文工程**（可检索、可验证的项目知识）才是决定稳定性的底盘。工程化 AI 的目标，不是追求单次惊艳的生成，而是把可复现、可审计、可协作的人机分工写进日常开发节奏。本文从人机边界、上下文沉淀与流程约束三个维度，整理我们在前端场景中的实践思路，供同行交流与复盘。

## 一、核心理念：人机协作的边界

### 1.1 AI 的能力边界

在界面与工程实现层面，AI 尤其擅长输入输出边界清晰的任务：依据既有接口契约生成调用与状态流转、在明确风格约束下完成样式与布局调整、将重复性机械劳动（样板代码、测试骨架、文档化结构）从人手中剥离。当任务可以被拆解为「可引用的规范 + 可对照的示例 + 明确的完成定义」时，模型往往能以较高效率给出可用初稿。相对地，复杂业务规则的取舍、跨版本的历史决策、涉及身份与权限的安全敏感路径，仍然需要工程师承担定义者与守门人的角色。AI 可以提出多种实现路径，但**谁为最终行为背书**必须清晰：模糊地带若交给模型「自由发挥」，成本会在联调与线上问题中加倍偿还。将能力边界写进团队共识，不是限制创新，而是把创造力集中在真正需要判断力的环节。

### 1.2 代码所有权

我们坚持一条朴素原则：**凡进入主分支的代码，无论是否经 AI 辅助编写，其质量与后果均由提交者承担。** 这意味着 Review、测试与发布责任不因「机器写过」而稀释。AI 是加速器，不是责任转嫁对象；工程师对架构一致性、可读性、安全与可维护性仍负全责。实践中，这一原则会反向塑造使用方式：在提示中主动给出约束、在合并前按标准审视生成结果、在发现系统性偏差时更新规范与知识库，而不是个案打补丁。当缺陷或事故追溯时，讨论焦点应是流程与上下文是否充分，而非将责任推给工具。所有权清晰后，团队更容易形成「人机协作的纪律」，而不是「个人与模型之间的私下默契」。

## 二、上下文工程：让 AI 懂你的项目

### 2.1 问题根因

通用大模型并不携带你方代码库的私有记忆；会话之间也不会自动延续隐性上下文。若仅依赖临时粘贴的几段代码与口头描述，模型只能做「合理猜测」，而猜测在工程上等同于风险。更棘手的是，口头上下文会随人员流动与迭代而漂移，未落盘的决策最终只在聊天记录里「曾经正确过」。上下文工程的核心，是把**项目特有的决策、约束与结构**转化为机器可读、人类可维护的文本资产，使每一次协作都站在同一份事实来源之上。它不是把文档写得漂亮，而是把文档写得**可被检索、可被引用、可被校验**——让 AI 与人类共享同一套「地面真相」。

### 2.2 项目知识库（RepoWiki）

我们采用仓库内结构化知识库（RepoWiki）承载前端上下文：文档按类型拆分，避免长篇叙事淹没关键约束；写法遵循「AI 可读」原则——多用标题层级、表格与清单呈现接口与状态，减少隐喻与口号式描述。下表概括七类文档及其角色，便于团队按需建设与增量更新。

| 文档类型 | 角色与要点 |
|----------|------------|
| 项目概述 | 模块边界、技术选型摘要、与上下游关系 |
| 技术规范 | 栈版本、命名与目录约定、性能基线 |
| 路由与页面结构 | 信息架构、权限与守卫、关键导航路径 |
| 组件库文档 | Props、事件、插槽等结构化说明，便于对照生成 |
| 状态管理 | Store 划分、数据流与副作用边界 |
| 业务知识 | 核心流程与领域概念，解释「为何如此设计」 |
| 测试用例 | 关键场景、期望行为与回归关注点 |

当知识库与代码同源演进时，AI 辅助修改才具备「就地取证」的基础，而不是在真空中编造。

### 2.3 AI 配置规范（.cursorrules）

在编辑器侧，我们通过项目级规则文件（如 `.cursorrules`）把**风格、禁区与协作习惯**固化为默认上下文。其典型覆盖范围包括：语言与框架偏好、组件与样式约定、测试与类型检查的期望、对安全与隐私的硬性约束（例如敏感信息不得写入提示或日志）、以及希望 AI 默认遵守的交互边界（何时应拆分任务、何时应请求更多上下文）。规则应与知识库相互引用：前者定义行为边界，后者提供领域事实，二者缺一都会出现「守规矩但不懂业务」或「懂业务但越界」的失衡。规则文件的价值在于减少「每次对话重复声明」的摩擦，并把团队共识前置到生成之前。它是上下文工程的延伸：不仅告诉模型「项目里有什么」，也明确「在这里应该怎样工作」。

## 三、约束工程：规范 AI 的开发流程

### 3.1 Spec-Driven Development（SDD）

若缺少 upfront 的规格说明，AI 编码容易陷入「看似完成、实则偏离」的陷阱：界面像样，状态与异常路径却未对齐需求。Spec-Driven Development 强调在动手写实现之前，先把**可讨论、可确认的规格**落到文档中，再驱动编码与验收。流程上可概括为：需求澄清 → 形成规格并与干系人对齐 → 在规格约束下实现与重构 → 交付后同步更新规格，避免文档与代码分道扬镳。工具层面，我们选用轻量的 OpenSpecs 路线：相较笨重的全流程平台，轻量工具更易于与现有 Git 工作流结合，降低协作门槛，使规格真正成为「大家愿意写、愿意改」的日常对象，而不是应付检查的附属物。SDD 的本质，是用可读规格为 AI 划定跑道；跑道清晰，生成才可控。

### 3.2 代码 Review 标准

AI 产出在 Review 维度上有其典型风险：幻觉式 API 或依赖、类型宽松导致的隐性契约漂移、快乐路径覆盖而忽略边界、以及安全与权限上的「看起来对」。我们采用**分级检视**：阻断项聚焦在正确性、安全与合规——必须与需求与规格一致，敏感路径与数据流需可证伪；建议项聚焦在可读性、一致性与长期维护成本——命名、抽象粒度、重复模式等可在合入后持续打磨。分级的好处是既不因完美主义阻塞交付，也不让高风险问题借「AI 写的」蒙混过关。Review 标准与 SDD、知识库联动：规格与文档越是可靠，Review 越能从「猜意图」转为「对照验收」。

## 四、实践效果

在引入仓库知识库、规格驱动节奏与分级 Review 之后，团队在前端交付上观察到几类可重复的收益。第一，**需求到实现的对齐成本下降**：规格与 RepoWiki 提供了共同参照，产品、设计与工程在合入前更容易对「完成定义」达成一致，从而减少返工与误解性合并。第二，**AI 辅助的可预测性提升**：上下文前置使生成结果更贴近项目约束，个案「救场式」提示与事后纠偏的频率显著降低。第三，**质量讨论更可审计**：阻断项与建议项分层后，风险暴露在流程早期，知识库与规格随代码更新，避免文档腐烂为无人敢信的摆设。上述变化并非来自单一工具，而来自把人与 AI 的分工写进工程习惯。

## 五、持续演进

AI 工具与模型能力仍在快速迭代，任何「一次性规范」都会过时。我们更看重**可演进的机制**：当新失败模式出现，将其沉淀为规格模板、规则条目或案例说明；当业务架构调整，优先更新知识库与 Specs，再放开大规模生成。案例库不必追求宏大叙事，**可复现的失败与修复路径**往往比成功故事更有迁移价值。对外分享的意义在于交换这类机制性经验，而非堆砌工具名单。若贵团队也在探索人机协作的工程化路径，欢迎就边界、上下文与约束实践展开对话，共同完善行业可参考的常识基线。
