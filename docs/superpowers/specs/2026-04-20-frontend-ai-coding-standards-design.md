## 一、前言

如何让 Agent 稳定且高质量地完成开发任务，是时下最热门的 AI 话题。当前业内的主流观点是，在 Coding 领域，除了提升模型自身的能力之外，外围的**约束工程**与**上下文工程**才是更能决定 Agent 是否能稳定且高质量完成任务的必要条件。

在我们日常使用 Agent 进行开发的过程中，实际上大家已经在做这些事情了：

- 编写提示词时，给 Agent 提供足够多的上下文、解释业务需求和项目相关知识；
- 开发之前通过工具让 Agent 生成开发计划、Specs，然后逐一开发；
- 在 Agent 开发完成后，让 Agent 执行测试用例，自动修正错误逻辑。

因此，我们希望将前端开发流程沉淀为一套可复用的规范，与后端、算法规范共享同一框架骨架，核心包含两个方面：

1. **项目知识库（RepoWiki）**：将 Agent 所需的上下文全部维护在代码仓库中，让 Agent 更懂我们的业务和技术约束。
2. **SDD（Spec-Driven Development）**：明确以 SDD 作为开发规范，让 Agent 开发流程更规范，消除模糊点，提升开发准确性。

## 二、整体架构

采用**共享内核 + 双层输出**模式：

```
┌─────────────────────────────────────────────┐
│              内容中台（共享内核）              │
│  核心原则 │ 标准流程 │ 技术栈实践 │ 案例库   │
└──────────────────┬──────────────────────────┘
                   │
        ┌──────────┴──────────┐
        ▼                     ▼
  📘 对外白皮书           📗 内部 Playbook
  方法论 + 精选案例       Checklist + 模板 + 标准
  受众：行业同行          受众：内部研发团队
```

**内容只写一次，两种形式输出：**

- **对外白皮书**：宏观方法论与理念阐述，精选脱敏案例，适合对外展示；
- **内部 Playbook**：可执行 Checklist、Prompt 模板、Rules 配置、Review 标准，新老员工直接可用。

## 三、内容中台四模块

### 3.1 核心原则（三人共定）

- AI 使用边界（什么场景适合用 / 什么不能依赖 AI）

- 安全与隐私红线（不向 AI 提交敏感数据、密钥等）
- 代码质量责任归属（AI 生成不免除 Review 责任）

- AI 生成代码 Review 标准与质量门禁

### 3.3 技术栈实践（前端负责）

见第四节详述。

### 3.4 案例库（三人各自沉淀）

- ✅ 最佳实践案例（含前端具体操作截图与 Prompt）

## 四、前端技术栈实践（5 个子模块）

> 主要工具：**Cursor**（AI IDE）  
> 主要框架：**Vue 3 / Nuxt**

### 4.1 Cursor 配置规范

**目标：** 统一团队 Cursor 工具层配置，减少个人差异。

内容包括：
- `.cursorrules` / Rules 文件模板（前端项目通用版 + Vue/Nuxt 专项版）
- MCP 工具配置建议（哪些 MCP 值得接入、如何配置）
- Cursor 项目级设置最佳实践（`@docs` 引用、ignore 配置等）

**验收标准：** 输出可复用的 Rules 模板文件，团队成员可一键配置。

---

### 4.2 RepoWiki 规范（前端版）

**目标：** 在每个前端代码仓库中维护一份结构化知识库，解决"AI 不懂项目上下文"的核心痛点。

**文件结构：**

```
repowiki/zh/content/
├── INDEX.md                    # RepoWiki 索引与使用说明
├── 01-项目概述.md
├── 02-技术规范/
│   ├── 技术栈.md               # Vue/Nuxt 版本、UI 库、状态管理工具
│   ├── 开发规范.md             # 命名规范、组件规范、CSS 规范
│   └── 性能要求.md
├── 03-路由与页面结构/
│   └── 路由结构.md             # 页面层级、路由守卫、权限控制
├── 04-组件库文档/
│   └── 核心组件API.md          # Props / Emits / Slots 定义（对 AI 友好格式）
├── 05-状态管理/
│   └── Pinia-Stores.md         # Store 结构、数据流说明
├── 06-业务知识/
│   ├── 核心流程.md
│   └── 领域概念.md
└── 07-测试用例/
    └── 测试用例.md             # E2E 场景、组件测试用例、预期结果
```

**撰写原则：**
- 所有文档以"AI 能读懂并直接使用"为标准，避免面向人类的叙事风格；
- 组件 API 文档使用结构化格式（表格或 JSON Schema）；
- 业务知识重点描述"为什么"而非"是什么"，帮助 AI 理解决策背景。

**验收标准：** 前端核心业务仓库均需生成一份 RepoWiki，提交到 main 分支。

---

### 4.3 SDD 规范（前端版）

**目标：** 以 Spec-Driven Development 作为前端开发规范，开发前先生成 Specs，减少 Agent 的自由发挥空间。

**工具选型：OpenSpecs**

选择原因：
- 流程简洁清晰，文档结构轻量，适合已拆解细化任务的团队协作场景；
- 开源工具，社区活跃；
- 与后端规范保持统一。

**前端 Specs 生成流程：**
1. 对存量仓库，先根据当前代码生成一份全量 Specs（作为单独任务）；
2. 新需求开发前，先生成页面级 / 功能级 Specs；
3. 开发完成后，Specs 与代码同步更新。

**验收标准：** 前端核心仓库负责人需对每个仓库生成一份全局 Specs，描述当前所有页面与功能，作为后续迭代的基础。

---

### 4.4 前端 Prompt 模板库

**目标：** 沉淀高频使用场景的 Prompt 模板，减少重复摸索成本。

覆盖场景：

| 场景 | Prompt 模板说明 |
|------|----------------|
| 组件生成 | 基于设计稿 / 需求描述生成 Vue SFC |
| 页面重构 | 保持功能不变，优化代码结构 |
| 接口对接 | 根据 API 文档生成数据请求与状态处理 |
| 样式调整 | 基于现有组件调整响应式样式 |
| 单测生成 | 为组件生成 Vitest 单元测试 |
| 调试分析 | 提供错误日志，让 AI 定位并修复问题 |

每个模板包含：**模板结构 + 上下文注入方式 + 约束语 + 使用示例**。

---

### 4.5 前端 Review 要点

**目标：** 明确 AI 生成前端代码的 Review 标准，避免质量参差不齐。

**Review Checklist：**

**功能正确性**
- [ ] 业务逻辑与需求描述一致
- [ ] 边界条件与异常状态处理完整
- [ ] 接口调用参数与 API 文档匹配

**代码质量**
- [ ] 组件职责单一，粒度合理
- [ ] Props / Emits 类型定义完整
- [ ] 无硬编码字符串（需走 i18n）

**性能**
- [ ] 无不必要的全量重渲染
- [ ] 大列表使用虚拟滚动
- [ ] 图片 / 资源已做懒加载处理

**安全**
- [ ] 无 XSS 注入风险（v-html 使用需审查）
- [ ] 敏感数据不输出到控制台
- [ ] 权限控制逻辑未被 AI 遗漏

**可维护性**
- [ ] 命名符合团队规范
- [ ] 无 AI 常见幻觉（如引用不存在的方法 / 库）
- [ ] 复杂逻辑有必要的注释

## 五、推荐开发流程

```
需求确认
   ↓
生成 / 更新 Specs（OpenSpecs）
   ↓
AI 参考 RepoWiki + Specs 进行编码（Cursor）
   ↓
按 Review Checklist 进行代码审查
   ↓
合并代码
   ↓
同步更新 Specs + RepoWiki   ← 完整的一次改动
```

> 更新完 Specs 和 RepoWiki 才算是一次完整的代码改动。
